Trade-offs: Fixed Pipeline vs Orchestrator

Fixed Pipeline

Pros

  • Less error-prone & predictable: Predefined input/output contracts, same sequence every time, easier to debug
  • Reliable: Well-tested execution path
  • Clear expectations: Easier for users to form expectation model of what it's doing

Cons

  • Rigid: Cannot adapt to different user needs
  • Limited: One-size-fits-all approach
  • Hard to modify: Changes require pipeline redesign

Orchestrator

Pros

  • Very flexible & dynamic: Adapts to user requests, can skip or repeat steps as needed
  • Extensible: Easy to add new tools
  • User-driven: Responds to natural language

Cons

  • Error-prone: Agent execution loop can fail
  • Unpredictable: Different paths each time, harder to debug
  • Unclear capabilities: User may not know what's possible

The Challenge: Design tools, data representations, and execution loops intelligently to get flexibility without sacrificing reliability

Making Frontend Actions Less Error-Prone

The Problem: Sequential Operations

Agent must track state and do math between calls:

1. addClipToTimeline("I", position: 0)        // Agent thinks: "Timeline = 150ms"
2. addClipToTimeline("hate", position: 150)    // Agent thinks: "Timeline = 570ms" 
3. addClipToTimeline("glaze", position: 570)   // Math error = broken timeline

Problem: Agent prone to calculation errors and state tracking mistakes

Better Approach: Batch Operations

Agent declares intent, frontend handles execution:

addMultipleClipsToTimeline({
  clips: [
    { fileId: "video-1", trimStartMs: 1200, trimEndMs: 1350 },  // "I"
    { fileId: "video-1", trimStartMs: 45200, trimEndMs: 45620 }, // "hate"  
    { fileId: "video-1", trimStartMs: 67800, trimEndMs: 68200 }  // "glaze"
  ]
})

Benefit: Deterministic code handles complex positioning logic, reducing errors
